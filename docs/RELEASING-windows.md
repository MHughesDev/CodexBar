# Releasing CodexBar for Windows

## 1. Prerequisites

| Requirement | Notes |
|---|---|
| **Swift 6.x for Windows** | Download from [swift.org/download](https://www.swift.org/download/). Installer adds Swift to `Program Files\Swift`. |
| **.NET 8 SDK** | `winget install Microsoft.DotNet.SDK.8` or from [dot.net](https://dotnet.microsoft.com/download). |
| **Windows 10 SDK (≥ 10.0.19041)** | Install via Visual Studio Installer → "Desktop development with C++" → "Windows 10 SDK". Required for `signtool.exe` and MSIX tooling. |
| **Windows App SDK** | Installed as a NuGet dependency by the shell project; no separate step needed for development. |
| **Authenticode cert** | OV cert works; EV cert bypasses SmartScreen warm-up period and is strongly preferred for public releases. Obtain from DigiCert, Sectigo, or equivalent CA. For CI, use **Azure Trusted Signing** (cheaper, no hardware token required). |
| **`gh` CLI** | For uploading release assets: `winget install GitHub.cli`. |

## 2. Build steps

Run these in order. Each step produces artifacts consumed by the next.

### 2a. Engine (Swift CLI)

```powershell
swift build -c release --product codexbar
```

The output binary is at `.build\release\codexbar.exe`. Collect the Swift runtime DLLs alongside it — they are in `%ProgramFiles%\Swift\Toolchains\<version>+Asserts\usr\bin`:

```
swiftCore.dll  swiftCRT.dll  Foundation.dll  FoundationNetworking.dll
FoundationXML.dll  dispatch.dll  BlocksRuntime.dll  swiftDispatch.dll
swiftWinSDK.dll
```

The CI job `build-engine` in `.github/workflows/windows-release.yml` automates this collection.

### 2b. Shell (.NET / WinUI 3)

```powershell
dotnet publish windows/CodexBar.Shell/CodexBar.Shell.csproj `
    -c Release -r win-x64 --self-contained true `
    -o publish\shell
```

Self-contained publish bundles the .NET 8 runtime so users need nothing pre-installed.

### 2c. MSIX package (optional — requires Visual Studio + WAP project)

Open `windows/CodexBar.Package/CodexBar.Package.wapproj` in Visual Studio 2022 (with the "Windows application packaging" workload) and build in Release|x64. Alternatively, use `msbuild`:

```powershell
msbuild windows\CodexBar.Package\CodexBar.Package.wapproj `
    /p:Configuration=Release /p:Platform=x64 `
    /p:AppxPackageSigningEnabled=false
```

The unsigned `.msix` ends up under `windows\CodexBar.Package\AppPackages\`.

## 3. Signing

Run `windows\sign-release.ps1` against every binary and the final package.

### With an Authenticode certificate from the Windows cert store

```powershell
# Sign the engine binary
.\windows\sign-release.ps1 -Target .build\release\codexbar.exe -Thumbprint <SHA1>

# Sign the shell executable
.\windows\sign-release.ps1 -Target publish\shell\CodexBar.exe -Thumbprint <SHA1>

# Sign the MSIX (publisher in Package.appxmanifest must match the cert Subject)
.\windows\sign-release.ps1 -Target CodexBar-1.0.0-win-x64.msix -Thumbprint <SHA1>
```

### With Azure Trusted Signing (CI recommended)

```powershell
.\windows\sign-release.ps1 -Target CodexBar-1.0.0-win-x64.msix -UseAzureTrustedSigning
```

Set `AZURE_TENANT_ID`, `AZURE_CLIENT_ID`, `AZURE_CLIENT_SECRET` (or use managed identity), and `ATS_DLIB_PATH` pointing to the restored `Azure.CodeSigning.Dlib.dll`. Create `windows\ats-metadata.json` per the [Azure Trusted Signing docs](https://learn.microsoft.com/azure/trusted-signing/how-to-signing-integrations).

## 4. Publish

### 4a. Update the WinSparkle appcast

Fill in the template at `windows/appcast-template.xml`:

```powershell
$version  = '1.0.0'
$pubDate  = (Get-Date -Format 'ddd, dd MMM yyyy HH:mm:ss K')
$msixPath = "CodexBar-$version-win-x64.msix"
$size     = (Get-Item $msixPath).Length
# DSA signature: generated by WinSparkle's sign_update tool against the private key
$dsaSig   = (& sign_update $msixPath $env:WINSPARKLE_PRIVATE_KEY)

(Get-Content windows\appcast-template.xml) `
    -replace '\$\{VERSION\}',  $version `
    -replace '\$\{PUB_DATE\}', $pubDate `
    -replace '\$\{SIZE\}',     $size `
    -replace '\$\{DSA_SIG\}',  $dsaSig |
  Set-Content appcast-windows.xml
```

Commit and push `appcast-windows.xml` **before** publishing the GitHub release so the URL is live when clients check for updates.

### 4b. Push the tag and let CI release

```powershell
git tag v1.0.0
git push origin v1.0.0
```

The `windows-release.yml` workflow triggers on `v*` tags, builds the portable zip, and uploads it to the GitHub release via `softprops/action-gh-release`. Manual runs (via `workflow_dispatch`) upload as workflow artifacts instead.

### 4c. Upload the MSIX manually (if not yet automated)

```powershell
gh release upload v1.0.0 `
    CodexBar-1.0.0-win-x64.msix `
    CodexBar-1.0.0-win-x64.msix.sha256 `
    --clobber
```

## 5. winget submission

winget pulls packages from the community manifest repo [microsoft/winget-pkgs](https://github.com/microsoft/winget-pkgs).

1. Fork `microsoft/winget-pkgs`.

2. Create the manifest directory:
   ```
   manifests/c/CodexBar/CodexBar/<version>/
   ```

3. Add three YAML files following the [winget manifest schema](https://github.com/microsoft/winget-pkgs/blob/master/doc/manifest/schema/):

   - `CodexBar.CodexBar.installer.yaml` — points to the signed `.msix` on GitHub Releases; set `InstallerType: msix`.
   - `CodexBar.CodexBar.locale.en-US.yaml` — display metadata (description, license, tags).
   - `CodexBar.CodexBar.yaml` — top-level manifest referencing the two above.

4. Use the `wingetcreate` tool to generate and validate the manifests:
   ```powershell
   wingetcreate new https://github.com/MHughesDev/CodexBar/releases/download/v1.0.0/CodexBar-1.0.0-win-x64.msix
   ```

5. Open a PR against `microsoft/winget-pkgs`. The automated pipeline validates the manifest; a human reviewer approves it.

For subsequent releases, run `wingetcreate update` or open a new PR with the bumped version directory.

## 6. Portable ZIP

The CI workflow assembles `CodexBar-{version}-win-x64.zip` containing:

```
codexbar.exe          ← Swift engine
swiftCore.dll         ┐
Foundation.dll        │ Swift runtime DLLs
...                   ┘
CodexBar.exe          ← WinUI 3 shell
(shell dependencies)
```

Users who want the CLI only can run `codexbar.exe` standalone; the DLLs must stay in the same directory. No installer required.

## Checklist

- [ ] Bump version in `Sources/CodexBarCLI/Version.swift` (or wherever `VERSION` is set) and `windows/CodexBar.Package/Package.appxmanifest`.
- [ ] Run `swift test --parallel` and `dotnet test` — zero failures.
- [ ] Build engine (`swift build -c release --product codexbar`).
- [ ] Build shell (`dotnet publish ... --self-contained`).
- [ ] Sign all binaries and the MSIX with `windows\sign-release.ps1`.
- [ ] Generate updated `appcast-windows.xml` with correct size and DSA signature; push to the feed URL before the tag.
- [ ] Push tag `v<version>` — CI builds and uploads the portable zip.
- [ ] Upload signed MSIX to the GitHub release (`gh release upload`).
- [ ] Verify enclosure URL resolves (`curl -I <msix-url>`).
- [ ] Submit or update the winget manifest PR.
- [ ] Smoke-test: download the zip, run `codexbar.exe --version`, run `CodexBar.exe`.
- [ ] Confirm WinSparkle "Check for Updates" finds the new release in a previous build.
